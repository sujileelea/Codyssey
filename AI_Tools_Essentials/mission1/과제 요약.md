# 1. 미션 요약

> 같은 모델이라도 "그냥 물어보는 것"과 "제대로 시키는 것"의 격차를 직접 체감하는 것이 출발점

- LLM의 작동 원리(확률적 생성·환각)를 이해하고, 프롬프트 엔지니어링으로 텍스트 결과물을 정교하게 제어
- 3종 이상의 LLM을 동일 과업에 투입해 평가 축·근거로 최적 모델을 선정
- 페르소나 기반 시스템 프롬프트를 Few-shot·단계적 추론 유도로 v1→v2 개선
- 10턴 이상 대화에서 문맥을 유지하고 환각을 체계적으로 검증

# 2. 과제 목표
> 수료 후 스스로 설명할 수 있다!

- LLM이 "확률적 생성" 기반으로 동작하며, 왜 환각이 발생하는지
- Zero-shot / Few-shot / 단계적 추론 유도 기법의 차이를 구분하고 과업에 맞게 적용했는지
- 3종 이상 LLM 비교 시 평가 축·근거 선택 이유
- 페르소나·시스템 프롬프트가 결과 품질(일관성/안전성/형식 준수)에 미치는 영향을 설명할 수 있는지
- 생성 결과를 테스트로 검증(특히 사실 오류)하고, 프롬프트를 반복 개선하는 과정을 재현 가능한지

## 4-2. 시스템 설계 문서

- [ ] 문제 정의(업무 자동화 과업 1개 선택)
- [ ] 타겟 사용자 / 입력 데이터 형태 / 출력 규격
- [ ] 재사용 가능한 입력 템플릿 1개 이상
- [ ] 페르소나: 이름 / 역할 / 전문 분야 / 말투 / 금지 사항 / 우선순위
- [ ] 시스템 프롬프트 4요소: 역할+목표 / 출력 형식 규칙 / 안전장치(모르면 모른다·확인 질문) / 사실·정책·수치 처리 규칙
- [ ] Few-shot 예시 3개 이상 (1개는 모호한 입력 → 되묻기 케이스)
- [ ] 단계적 추론 유도 v1 vs v2 비교 + "최종 답변은 요약/근거 중심" 노출 규칙
- [ ] 환각 검증 설계: 검증 질문 5개 이상 + 기대 정답·허용 오차·참고 근거 + Pass/Fail 기준
- [ ] 개선 이력 v1→v2 + 최종 프롬프트 전문

## 4-3. 실행 로그

- [ ] 시나리오 설명
- [ ] 10턴 이상 대화 전문
- [ ] 변동 상황 1개 이상: 조건 변경(톤/대상/정책) 또는 추가 정보 뒤늦게 제공
- [ ] 문맥 유지 검증(일관 유지 + 변경분만 반영 → Pass 판정)
- [ ] 문제 지점(환각/문맥 끊김)과 수정 결과 요약
- [ ] 원본 로그(전문) 별도 파일 보존

# 5. 핵심 개념 정리

## 5-1. 프롬프트 기법

- **Zero-shot**: 예시 없이 지시만으로 수행.
- **Few-shot**: "좋은 입력 → 좋은 출력" 예시를 함께 제공해 형식·품질을 유도.
- **단계적 추론 유도(Reasoning)**: 요구사항 정리 → 누락 질문 → 초안 작성처럼 단계를 밟게 함. 단, 최종 답변에는 장문 추론을 노출하지 않고 핵심 근거 중심으로 요약.

## 5-2. 환각(Hallucination) 정의 — 본 미션 기준

> 사실/수치/정책/계산처럼 정답 또는 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변.

- 창의적 글쓰기(소설/카피)는 환각으로 보지 않음 — 단 "허구" 명시 또는 세계관 규칙 준수 필요.
- **Pass**: (A) 정답 + 근거 제시, 또는 (B) 모르면 "모른다/확인 필요" 명시 + 확인 절차 제안.
- **Fail**: 틀린 사실을 단정(근거·불확실성 표기 없음), 또는 전제 불명확한데 확인 질문 없이 임의로 사실 생성.

# 7. 제약 사항 (감점 주의)

- **재현성**: 모델명 / 요금제 / 사용 채널 / 사용 날짜(YYYY-MM-DD) / 주요 설정을 제출물에 반드시 기록. 무료 버전은 제한점 1줄 명시.
- **안전·윤리**: 폭력적/선정적 콘텐츠 금지. 개인정보(실명·전화·주소·계정정보)를 프롬프트/로그에 직접 포함 금지. 사실/정책/수치는 "근거·확인 필요" 규칙 준수.
- **제출물 편집**: 대화 로그를 그대로 덤프하지 말고 섹션/번호/요약으로 정리. 단, 원본 로그(전문)는 별도 파일로 보존.
