# 과제 요약

## 문서 목적

이 문서는 projectB 과제가 무엇을 요구하는지, 최종 산출물이 무엇인지, 팀이 공통으로 이해해야 할 핵심 목표와 제출 기준을 정리한 문서이다.

## 한 문장 정의

projectB의 과제는 노코드 자동화 플랫폼을 이용해 `뉴스 수집 -> 필터링 -> AI 요약 -> Notion 저장`의 전 과정을 하나의 자동화 워크플로우로 연결하는 것이다.

## 과제 핵심 요약

- RSS 피드에서 최신 기술 뉴스를 자동 수집한다.
- 사전에 정의한 주제 필터링 기준에 맞는 기사 1건을 선택한다.
- 생성형 AI를 사용해 기사 내용을 3줄 이내로 요약한다.
- 요약 결과를 노션 등 협업 도구의 데이터베이스에 자동 저장한다.
- 스케줄 트리거, 주제 필터링, AI 요약, 저장, 중복 방지, 오류 처리까지 포함한 자동화 파이프라인 전체를 설계하고 검증한다.
- 전체 과정은 사람의 수동 개입 없이 자동으로 완료되어야 한다.

## 과제 목표

- RSS 피드와 자동화 툴을 활용해 `데이터 수집 -> AI 가공 -> 저장`으로 이어지는 자동화 워크플로우를 설명할 수 있다.
- 생성형 AI 툴을 활용해 텍스트 요약을 자동화하는 방법을 설명할 수 있다.
- 스케줄 트리거와 에러 처리를 포함한 안정적인 자동화 시스템의 설계 원리를 설명할 수 있다.

## 과제 범위

이번 과제는 단순히 뉴스를 가져와 저장하는 기능만 만드는 것이 아니다. 데이터 수집, 정제, AI 가공, 저장, 품질 검수, 예외 처리까지 하나의 유기적인 파이프라인으로 연결해야 한다.

포함되는 핵심 범위는 다음과 같다.

- 정해진 시간에 자동 실행되는 스케줄 트리거
- RSS 피드 기반 뉴스 수집
- 제목, 링크, 본문, 발행일시 등 필수 데이터 추출
- 기술 뉴스 또는 선택 주제에 맞는 기사 필터링
- 생성형 AI 기반 3줄 이내 한국어 요약
- AI 요약 결과의 환각 방지와 품질 검수
- Notion 데이터베이스 필드 매핑
- 중복 저장 방지
- 실패 시 최대 2회 재시도
- 최종 실패 시 로그 기록, 스킵, 필요 시 알림
- API 키와 토큰 노출 방지

## Notion 데이터베이스 요구 설계

| 속성명 | 속성 타입 | 저장 내용 | 예시 |
| --- | --- | --- | --- |
| 제목 | Title | 뉴스 제목 | AI 에이전트의 미래와 전망 |
| 요약문 | Text | AI가 생성한 3줄 이내 요약 | 최신 AI 에이전트는... |
| 원문 링크 | URL | 기사 원문 URL | https://example.com/news/123 |
| 발행 일시 | Date | RSS에서 추출한 발행일시 | 2025-01-20 09:00 |
| 중복 방지 키 | Text | GUID 또는 원문 링크 해시 | `guid_123` 또는 `hash_url` |
| 감성 태그 | Select | 보너스 선택 시 긍정/부정/중립 | 긍정 |
| 썸네일 | Files & media / URL | 보너스 선택 시 생성 이미지 | 이미지 URL |

## AI 요약 요구사항

### 기본 요약 프롬프트

```text
아래 뉴스 본문을 한국어로 3줄 이내로 요약해줘.
원문에 없는 내용은 추가하지 말고, 핵심 사실과 시사점만 포함해줘.

제목: {{title}}
본문: {{content}}
원문 링크: {{url}}
```

### 요약 규칙

- [ ] 3줄 이내로 요약한다.
- [ ] 원문에 없는 내용을 추가하지 않는다.
- [ ] 과장된 해석이나 환각을 방지한다.
- [ ] 기사 1건당 요약 호출은 1회만 수행한다.
- [ ] 실패로 인한 재시도는 최대 2회까지만 허용한다.

## 자동 저장 요구사항

- 저장 대상: `Notion 데이터베이스 / 기타: ________`
- 저장 방식: `자동화 툴의 Notion 연동 모듈 / API / 기타: ________`
- 필드 매핑:
  - RSS 제목 -> 노션 `제목`
  - AI 요약 결과 -> 노션 `요약문`
  - RSS 링크 -> 노션 `원문 링크`
  - RSS 발행일시 -> 노션 `발행 일시`
  - GUID 또는 링크 해시 -> 노션 `중복 방지 키`

## 안정성 및 오류 처리 요구사항

| 오류 상황 | 처리 정책 | 선택 이유 |
| --- | --- | --- |
| RSS에서 뉴스가 수집되지 않음 | 로그 기록 후 해당 실행 회차 스킵 | 빈 데이터로 AI 호출이나 저장을 하지 않기 위해 |
| 필터 조건을 만족하는 뉴스가 없음 | 로그 기록 후 스킵 | 주제와 무관한 기사를 저장하지 않기 위해 |
| AI 요약 API 호출 실패 | 최대 2회 재시도 후 실패 로그 기록 | 일시적 장애는 복구하되 무한 호출 비용을 방지하기 위해 |
| 노션 저장 실패 | 최대 2회 재시도 후 실패 로그 및 알림 | 저장 누락을 확인할 수 있도록 하기 위해 |
| 필수 필드 누락 | 저장하지 않고 오류 로그 기록 | 불완전한 데이터가 DB에 쌓이는 것을 방지하기 위해 |
| 중복 기사 감지 | 저장하지 않고 스킵 | 동일 기사 중복 저장을 방지하기 위해 |

### 재시도 정책

- [ ] 재시도는 최대 2회까지만 허용한다.
- [ ] 재시도 후에도 실패하면 로그를 남기고 해당 항목을 스킵한다.
- [ ] 무한 루프 또는 과도한 API 호출이 발생하지 않도록 설정한다.

### 중복 방지 정책

- 중복 방지 키: `GUID / 원문 링크 해시 / 원문 URL / 기타: ________`
- 적용 방식: `저장 전 기존 DB에서 동일 키 검색 후, 동일 키가 있으면 저장하지 않음`
- 선택 이유: `동일 기사가 여러 번 저장되는 것을 방지하고 AI 요약 비용을 줄이기 위해`

## 보안 정책

- [ ] API 키와 토큰은 환경 변수 또는 비공개 저장소에 보관한다.
- [ ] README, 스크린샷, 시연 자료에 API 키와 토큰이 노출되지 않도록 마스킹한다.
- [ ] 자동화 툴 화면 캡처 시 인증 정보, 토큰, 시크릿 값이 보이지 않도록 처리한다.
- [ ] RSS 피드가 없는 사이트를 강제 크롤링하지 않는다.
- [ ] 저작권을 침해할 수 있는 방식으로 본문 전체를 무단 수집하지 않는다.
