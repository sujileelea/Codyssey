# 서비스 구현 설계

## 문서 목적

이 문서는 뉴스 요약 자동화 시스템을 실제 서비스 또는 자동화 시나리오로 구현하기 위한 구조, 모듈, 데이터 흐름, 프롬프트, Notion 매핑, 오류 처리, 테스트 기준을 정리한 설계 문서이다.

## 구현 목표

매일 설정된 시간에 RSS 피드로 기술 뉴스를 자동 수집하고, 필요한 기사만 필터링한 뒤, 생성형 AI로 3줄 이내 한국어 요약을 생성하고, 정규화된 결과를 Notion 데이터베이스의 올바른 속성에 자동 저장한다.

서비스 구현의 핵심은 다음이다.

- 노코드 자동화 플랫폼으로 전체 과정을 연결한다.
- RSS에서 필요한 필드를 안정적으로 추출한다.
- 기술 뉴스 또는 선택 주제에 맞는 기사만 AI 단계로 넘긴다.
- AI 요약은 3줄 이내, 원문 기반, 환각 방지 원칙을 따른다.
- Notion에는 제목, 요약문, 원문 링크, 발행 일시, 중복 방지 키를 매핑한다.
- 동일 기사는 중복 저장하지 않는다.
- 오류 발생 시 최대 2회까지만 재시도한다.
- 최종 실패 시 로그를 남기고, 필요하면 알림을 보낸다.
- API 키와 토큰은 문서와 스크린샷에 노출하지 않는다.

## 목표 워크플로우 구조

```text
[1] 스케줄 트리거
    - 매일 설정 시간 자동 실행
    - 타임존: Asia/Seoul
        ->
[2] RSS 수집
    - 기술 뉴스 RSS 피드에서 최신 항목 수집
    - 제목, 링크, 본문, 발행일시 추출
        ->
[3] 주제 필터링
    - 사전 정의한 키워드/태그 기준으로 기사 필터링
    - 조건을 만족하는 기사 1건 선택
        ->
[4] AI 요약
    - 생성형 AI 모델 호출
    - 3줄 이내 요약 생성
    - 기사 1건당 요약 호출 1회 원칙 적용
        ->
[5] 품질게이트
    - 요약문 길이, 원문 근거, 필수 필드, 중복 여부 확인
        ->
[6] 노션 DB 저장
    - 제목, 요약문, 원문 링크, 발행일시, 중복 방지 키 저장
        ->
[7] 예외 처리
    - 실패 시 최대 2회 재시도
    - 최종 실패 시 로그 기록, 스킵, 필요 시 알림
```

## 현재 Make 시나리오 기반 구현 구조

현재 Make에는 다음 구조가 구현되어 있다.

```text
BBC RSS 뉴스 수집
    -> Make AI Toolkit으로 기술 뉴스 여부 판단 및 3줄 요약
    -> AI 답변에 SKIP이 포함되지 않을 때만 통과
    -> Notion 뉴스 저장 데이터베이스에 저장
```

현재 설정값은 다음과 같다.

- 자동화 툴: Make
- RSS 피드: `http://feeds.bbci.co.uk/news/rss.xml`
- RSS 수집 개수: 실행 1회당 최대 5개의 새 기사
- 실행 주기: 15분마다
- AI 모듈: Make AI Toolkit
- 현재 시나리오에 기록된 AI 모델: OpenAI GPT-5.5
- 필터 이름: `기술 뉴스만 저장`
- 필터 조건: AI의 `Answer`에 `SKIP`이 포함되지 않을 때만 통과
- Notion DB 이름: `뉴스 저장`
- 현재 저장 필드:
  - 이름: RSS 기사 제목
  - AI 3줄 요약: AI 요약문
  - URL: 원본 기사 링크

## 목표 설계와 현재 구현의 차이

| 항목 | 현재 Make 시나리오 | 목표 설계 |
| --- | --- | --- |
| 실행 주기 | 15분마다 실행 | 매일 정해진 시간 실행 또는 15분 실행 선택 이유 문서화 |
| RSS 출처 | BBC 전체 뉴스 RSS | 기술 뉴스 전용 RSS 또는 AI 전 키워드 필터 추가 |
| 필터링 | AI가 `SKIP` 판단 | AI 전 키워드 1차 필터 + AI 구조화 분류 권장 |
| AI 요약 | 기술 뉴스면 3줄 이내 한국어 요약 | 3줄 이내, 원문 기반, 환각 방지, 실패 시 최대 2회 재시도 |
| Notion 저장 | 제목, 요약, URL | 제목, 요약문, 원문 링크, 발행 일시, 중복 방지 키 |
| 중복 방지 | 현재 설명에는 없음 | GUID, 원문 URL, 링크 해시 기반 중복 검색 후 저장 |
| 오류 처리 | 현재 설명에는 명확하지 않음 | RSS, AI, Notion 실패별 최대 2회 재시도와 로그/알림 |
| 테스트 증빙 | 7일 271 크레딧, 2026-06-23 실행 기록 | 정상 흐름과 예외 흐름 테스트 체크리스트 문서화 |

## Make 모듈 설계

### 1. Scheduler

- 목적: 워크플로우를 정해진 시간에 자동 실행한다.
- 요구 조건:
  - 실행 시간: `매일 __:__`
  - 타임존: `Asia/Seoul`
- 현재 상태:
  - 15분마다 자동 실행으로 설정되어 있다.
- 설계 판단:
  - 과제 요구사항에 맞추려면 매일 정해진 시간 실행으로 변경한다.
  - 15분 실행을 유지한다면 `더 빠른 최신 뉴스 수집을 위해 15분마다 실행`한다고 문서화한다.

### 2. RSS 수집 모듈

- 목적: RSS 피드에서 최신 뉴스 항목을 가져온다.
- 현재 RSS 피드:
  - `http://feeds.bbci.co.uk/news/rss.xml`
- 현재 실행 1회당 수집 개수:
  - 최대 5개의 새 기사
- 추출 필드:
  - 제목
  - 링크
  - 본문 또는 설명
  - 발행일시
  - GUID가 있는 경우 GUID

### 3. 필수 필드 검사

AI 비용 낭비를 막기 위해 다음 필드가 없는 기사는 AI 단계로 넘기지 않는다.

- 제목
- 원문 링크
- 본문 또는 요약 가능한 설명

발행일시가 없을 경우 처리 정책을 정한다.

- 선택지 1: 저장하지 않고 오류 로그 기록
- 선택지 2: 수집 시각을 대체값으로 저장
- 권장: 과제 요구상 발행일시 저장이 필요하므로, 원본 발행일시가 없으면 로그를 남기고 스킵한다.

### 4. 키워드 기반 1차 필터

현재는 AI가 기술 뉴스 여부를 판단하지만, 비용 절감을 위해 AI 호출 전에 키워드 기반 필터를 두는 것이 좋다.

#### 필터링 주제

- 선정 주제: `예: AI`

#### 키워드 또는 태그 목록

최종 문서에는 팀이 실제 선택한 키워드를 작성해야 한다.

- `________`
- `________`
- `________`

현재 시나리오의 AI 분류 기준을 키워드 후보로 바꾸면 다음과 같다.

- 기술
- AI
- 소프트웨어
- 하드웨어
- 반도체
- 클라우드
- 사이버보안
- IT기업

#### 선택 이유 작성 기준

- 왜 이 주제가 시장 조사에 적합한지 작성한다.
- 왜 이 키워드 또는 태그가 해당 주제를 잘 대표하는지 작성한다.
- 너무 넓거나 좁은 필터가 되지 않도록 어떤 기준을 적용했는지 작성한다.

### 5. 중복 방지 검색

AI 요약 호출 전에 중복 검사를 먼저 수행하는 것이 가장 좋다.

#### 중복 방지 키 후보

- RSS GUID
- 원문 URL
- 원문 링크 해시

#### 권장 방식

1. RSS 항목에서 GUID가 있으면 GUID를 중복 방지 키로 사용한다.
2. GUID가 없으면 원문 URL을 사용한다.
3. URL 비교가 불안정하면 URL을 해시로 변환해 사용한다.
4. Notion DB에서 같은 중복 방지 키가 있는지 검색한다.
5. 이미 존재하면 AI 요약을 호출하지 않고 스킵한다.
6. 존재하지 않으면 AI 요약 단계로 진행한다.

#### 적용 이유

동일 기사가 여러 번 저장되는 것을 방지하고 AI 요약 비용을 줄이기 위해서이다.

### 6. AI 요약 모듈

#### 기본 프롬프트

```text
아래 뉴스 본문을 한국어로 3줄 이내로 요약해줘.
원문에 없는 내용은 추가하지 말고, 핵심 사실과 시사점만 포함해줘.

제목: {{title}}
본문: {{content}}
원문 링크: {{url}}
```

#### 현재 Make 시나리오 프롬프트 역할

```text
당신은 IT 뉴스 분류기입니다.
기사 제목과 내용을 읽고, 기술/AI/소프트웨어/하드웨어/반도체/클라우드/사이버보안/IT기업 관련 뉴스면 한국어 3줄 이내로 요약해 주세요.
정치, 경제, 스포츠, 연예, 교육, 일반 사회 뉴스는 SKIP 이라고만 출력하세요.
```

#### 권장 구조화 프롬프트

현재 `SKIP` 문자열 포함 여부로 필터링하는 구조는 간단하지만 취약하다. 더 안정적인 설계는 AI가 구조화된 값을 출력하도록 하는 것이다.

```text
당신은 IT 뉴스 분류기이자 뉴스 요약기입니다.
아래 기사 제목과 내용을 읽고 기술/AI/소프트웨어/하드웨어/반도체/클라우드/사이버보안/IT기업 관련 뉴스인지 판단하세요.

규칙:
1. 기술 뉴스가 아니면 is_tech를 false로 설정하고 summary는 빈 문자열로 둡니다.
2. 기술 뉴스이면 is_tech를 true로 설정하고 summary에 한국어 3줄 이내 요약을 작성합니다.
3. 원문에 없는 내용을 추가하지 않습니다.
4. 과장된 해석이나 추측을 넣지 않습니다.
5. 출력은 아래 JSON 형식만 사용합니다.

출력 형식:
{
  "is_tech": true,
  "summary": "3줄 이내 한국어 요약",
  "reason": "기술 뉴스로 판단한 근거"
}

제목: {{title}}
본문: {{content}}
원문 링크: {{url}}
```

#### 요약 규칙

- [ ] 3줄 이내로 요약한다.
- [ ] 원문에 없는 내용을 추가하지 않는다.
- [ ] 과장된 해석이나 환각을 방지한다.
- [ ] 기사 1건당 요약 호출은 1회만 수행한다.
- [ ] 실패로 인한 재시도는 최대 2회까지만 허용한다.

### 7. AI 결과 필터

#### 현재 방식

- AI의 `Answer`가 `SKIP`을 포함하지 않을 때만 다음 단계로 진행한다.

#### 권장 방식

- 구조화 출력에서 `is_tech = true`일 때만 다음 단계로 진행한다.
- `is_tech = false`이면 로그 기록 후 스킵한다.
- AI 응답 형식이 깨졌다면 최대 2회 재시도 후 실패 로그를 남긴다.

### 8. 요약본 정규화

AI 출력은 매번 형식이 조금씩 달라질 수 있다. 따라서 Notion 저장 전에 정규화한다.

정규화 항목은 다음과 같다.

- AI 요약본 앞뒤 공백 제거
- 불필요한 줄바꿈 정리
- 줄바꿈 기호 `\n` 통일
- 번호 매기기 방식 정리
- 불필요한 마크다운 문법 제거
- Notion DB가 받아들이는 텍스트 형식으로 변환
- 원문 링크 URL 형식 확인
- 발행일시를 ISO 8601 형식으로 변환
- 발행일시 예시: `YYYY-MM-DD`
- 저장 전 제목, 요약문, 원문 링크, 발행일시, 중복 방지 키가 모두 있는지 확인

### 9. 품질게이트

품질게이트는 AI 결과가 Notion에 저장되기 전에 적용한다.

검수 기준은 다음과 같다.

- 요약문이 3줄 이내인지 확인한다.
- 요약문이 원문에 근거하는지 확인한다.
- 원문에 없는 내용을 지어내지 않았는지 확인한다.
- 제목, 링크, 본문, 발행일시 등 필수 데이터가 누락되지 않았는지 확인한다.
- Notion DB 속성에 텍스트가 깨지지 않고 매핑되는지 확인한다.
- 동일 기사가 이미 저장되어 있지 않은지 확인한다.
- 기사 1건당 AI 요약 호출이 1회 원칙을 지키는지 확인한다.

### 10. Notion 저장 모듈

#### 저장 대상

- 저장 대상: `Notion 데이터베이스 / 기타: ________`
- 현재 Notion DB 이름: `뉴스 저장`
- 저장 방식: `자동화 툴의 Notion 연동 모듈 / API / 기타: ________`

#### 필드 매핑

- RSS 제목 -> 노션 `제목`
- AI 요약 결과 -> 노션 `요약문`
- RSS 링크 -> 노션 `원문 링크`
- RSS 발행일시 -> 노션 `발행 일시`
- GUID 또는 링크 해시 -> 노션 `중복 방지 키`

#### Notion 데이터베이스 설계

| 속성명 | 속성 타입 | 저장 내용 | 예시 |
| --- | --- | --- | --- |
| 제목 | Title | 뉴스 제목 | AI 에이전트의 미래와 전망 |
| 요약문 | Text | AI가 생성한 3줄 이내 요약 | 최신 AI 에이전트는... |
| 원문 링크 | URL | 기사 원문 URL | https://example.com/news/123 |
| 발행 일시 | Date | RSS에서 추출한 발행일시 | 2025-01-20 09:00 |
| 중복 방지 키 | Text | GUID 또는 원문 링크 해시 | `guid_123` 또는 `hash_url` |
| 감성 태그 | Select | 보너스 선택 시 긍정/부정/중립 | 긍정 |
| 썸네일 | Files & media / URL | 보너스 선택 시 생성 이미지 | 이미지 URL |

## 오류 처리 설계

| 오류 상황 | 처리 정책 | 선택 이유 |
| --- | --- | --- |
| RSS에서 뉴스가 수집되지 않음 | 로그 기록 후 해당 실행 회차 스킵 | 빈 데이터로 AI 호출이나 저장을 하지 않기 위해 |
| 필터 조건을 만족하는 뉴스가 없음 | 로그 기록 후 스킵 | 주제와 무관한 기사를 저장하지 않기 위해 |
| AI 요약 API 호출 실패 | 최대 2회 재시도 후 실패 로그 기록 | 일시적 장애는 복구하되 무한 호출 비용을 방지하기 위해 |

## 재시도 정책

- [ ] 재시도는 최대 2회까지만 허용한다.
- [ ] 재시도 후에도 실패하면 로그를 남기고 해당 항목을 스킵한다.
- [ ] 무한 루프 또는 과도한 API 호출이 발생하지 않도록 설정한다.

## 로그 및 알림 설계

최종 실패 시 다음 중 하나 이상의 방식으로 로그를 남긴다.

- Make 실행 로그
- 별도 Notion 오류 로그 데이터베이스
- Google Sheet 오류 로그
- Slack 또는 이메일 알림

로그에 포함할 정보는 다음과 같다.

- 실패 발생 시각
- 실패 단계
- 기사 제목
- 원문 링크
- 오류 메시지
- 재시도 횟수
- 최종 처리 결과

## 보안 설계

- [ ] API 키와 토큰은 환경 변수 또는 비공개 저장소에 보관한다.
- [ ] README, 스크린샷, 시연 자료에 API 키와 토큰이 노출되지 않도록 마스킹한다.
- [ ] 자동화 툴 화면 캡처 시 인증 정보, 토큰, 시크릿 값이 보이지 않도록 처리한다.
- [ ] RSS 피드가 없는 사이트를 강제 크롤링하지 않는다.
- [ ] 저작권을 침해할 수 있는 방식으로 본문 전체를 무단 수집하지 않는다.

## 구현 순서

1. Notion 데이터베이스 속성을 먼저 확정한다.
2. Make 스케줄러를 매일 정해진 시간 또는 문서화된 실행 주기로 설정한다.
3. RSS 수집 모듈에서 제목, 링크, 본문, 발행일시, GUID를 추출한다.
4. 필수 필드 누락 검사를 추가한다.
5. 키워드 기반 1차 필터를 추가한다.
6. Notion에서 중복 방지 키를 검색한다.
7. 중복이 없을 때만 AI 요약 모듈을 호출한다.
8. AI 출력은 3줄 이내 요약과 기술 뉴스 여부가 분리되도록 구조화한다.
9. AI 요약문을 정규화한다.
10. 품질게이트를 통과한 데이터만 Notion에 저장한다.
11. RSS, AI, Notion 단계별 에러 핸들러를 설정한다.
12. 실패 시 최대 2회 재시도 후 로그 또는 알림을 남긴다.
13. 정상 흐름과 예외 흐름 테스트를 수행한다.
14. 스크린샷과 Notion 저장 결과를 제출 문서에 첨부한다.
